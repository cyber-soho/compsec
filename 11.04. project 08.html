Part 1

Theory — The Landscape

WEIGHT × 1

1.1 Why Software Alone Is Never Enough

A common mistake among newcomers to cybersecurity is imagining that installing the right software solves the problem. It does not. Security software is one layer of what practitioners call defence in depth — the principle that no single control, no matter how good, should be the last line of defence.

The NIST (National Institute of Standards and Technology) Cybersecurity Framework organises controls into Identify, Protect, Detect, Respond, and Recover — software tools appear in every one of those categories, working alongside policy, people, and physical controls.

With that caution stated, computer security software matters enormously. A well-configured, properly monitored suite of tools raises the cost of an attack — in time, skill, and resources — to the point where many opportunistic attackers simply move on to an easier target.

1.2

The Eight Major Categories

🛡️
01 / ENDPOINT PROTECTION

EPP & EDR

Run directly on individual machines. Traditional AV uses signature-based detection; modern EPP adds heuristics and sandboxing. EDR continuously records endpoint activity for threat hunting.

EPP · EDR · AV
🔥
02 / NETWORK CONTROL

Firewalls & WAF

Packet-filtering → Stateful → NGFW (Next-Generation Firewall) → WAF (Web Application Firewall). Each generation understands more context about what is passing through it.

NGFW · WAF · HTTPS
🔍
03 / DETECTION & PREVENTION

IDS & IPS

IDS (Intrusion Detection System) alerts; IPS (Intrusion Prevention System) acts inline. Signature-based detection finds known attacks; anomaly-based detects novel ones but with more false positives.

IDS · IPS · NSE
🔐
04 / SECURE TRANSPORT

VPN Software

Creates encrypted tunnels across untrusted networks. IPsec (Internet Protocol Security) operates at network layer; TLS-based VPNs at transport layer. OpenVPN and WireGuard are widely deployed open-source implementations.

VPN · IPsec · TLS/SSL
📊
05 / LOG CORRELATION

SIEM Platforms

SIEM (Security Information and Event Management) aggregates logs from across infrastructure and correlates them. Central to any SOC (Security Operations Centre). Turns individual events into patterns.

SIEM · SOC
🎯
06 / OFFENSIVE ASSESSMENT

Vulnerability Scanners

Discover weaknesses before attackers do. Produce CVE (Common Vulnerabilities and Exposures) identifiers and CVSS (Common Vulnerability Scoring System) severity scores. Penetration frameworks verify exploitability.

CVE · CVSS · NVD
🔑
07 / CREDENTIAL HYGIENE

Password Managers

Reduce the most common source of account compromise. Well-designed managers derive keys using PBKDF2 (Password-Based Key Derivation Function 2) or Argon2 — never storing the master password directly.

PBKDF2 · Argon2
📤
08 / DATA PROTECTION

DLP Software

DLP (Data Loss Prevention) monitors data in motion, at rest, and in use to detect or block exfiltration. Uses pattern matching, document fingerprinting, and increasingly machine-learning classification.

DLP · ML

1.3 How Professionals Evaluate Security Software

When confronted with a purchasing or deployment decision, professionals rarely rely on vendor marketing. Two widely cited evaluation approaches:

  • Common Criteria (CC) — international standard ISO/IEC 15408 defining a framework for certifying products meet security requirements. Products receive an EAL rating from 1 to 7.
  • Independent testing organisations such as AV-TEST and AV-Comparatives run standardised benchmarks — detection rates, false positive rates, performance impact. Learn to read the methodology, not just the headline score.

Exercise A — Mapping the Software Ecosystem on a Live System

Objective · Gap analysis · Firewall inspection · Est. 90 min

Objective: Develop a clear picture of what security software is already present — or absent — on a system you control, and identify gaps relative to the eight-category baseline from Part 1.

  • 1
    Inventory running processes. Identify which, if any, relate to security functions (antivirus agents, firewall daemons, VPN clients, logging agents). Build a table: Process name / Purpose / Category / Running (Y/N). 
    # Linux — process list
ps aux
systemctl list-units --type=service
ss -tulnp

# Windows PowerShell
Get-Service
netstat -ano
  • 2
    Gap analysis. Compare your inventory against the eight categories in §1.2. For every category with no software representative, briefly explain what threat surface is left unaddressed. This feeds directly into your Page 3 report section.
  • 3
    Firewall rule inspection. Examine current firewall configuration. Identify (a) one rule that makes sense and explain why, and (b) one rule or gap you would improve and how. 
    # Linux — view iptables rules
sudo iptables -L -v -n
# or nftables
sudo nft list ruleset

# Windows
netsh advfirewall show allprofiles
🚀 Going Further Set up a minimal ufw (Uncomplicated Firewall) ruleset on a Linux VM and demonstrate, by attempting a connection from the host machine, that your rules behave as intended. Screenshot and describe the result.
🔎

Exercise B — Vulnerability Scanning on a Controlled Target

Objective · Tool selection · CVE lookup · CVSS scoring · Est. 90 min

Objective: Run a vulnerability scan against a system you own or a purpose-built vulnerable VM, interpret the output, and match findings to CVE (Common Vulnerabilities and Exposures) records.

Choose Your Target (one only)

TargetTypeWhere to get it
Metasploitable 2/3 Deliberately vulnerable Linux VM github.com/rapid7/metasploitable3
DVWA Damn Vulnerable Web Application — Docker github.com/digininja/DVWA
Your own VM Service you installed & understand Your own lab environment
  • 1
    Research and justify your scanning tool — spend 20 minutes exploring your options before running anything. Document why you chose the tool: ease of setup, scope of detection, documentation quality, community support.
    Nmap + NSE Scripts OpenVAS / Greenbone
  • 2
    Run the scan. Read what each flag does before running — do not scan blindly. 
    # Nmap with vulnerability scripts — read the manual first!
nmap -sV --script vuln <target-IP>
    Capture the full output. You will quote from it in your report.
  • 3
    Select THREE findings and for each one:
    • State the finding as the tool reports it.
    • Look up the CVE ID on the NVD (National Vulnerability Database).
    • Record the CVSS (Common Vulnerability Scoring System) score and what it tells you about severity.
    • Describe — in your own words — what class of harm this vulnerability could enable.
    • Suggest a remediation step.
🚀 Going Further Look up whether a patch or configuration change addressing one of your findings is documented in the vendor's security advisory. Link to it. Compare whether the vendor's described impact matches the NVD record.
📋

Exercise C — Evaluating an Endpoint Protection Solution

Objective · Criteria definition · Evidence gathering · Weighted scoring · Est. 60 min

Objective: Apply a structured evaluation methodology to a real tool, as a professional analyst would before recommending a purchase or deployment.

  • 1
    Select one tool to evaluate. Examples below — or find your own. Justify your selection.
    ClamAV (open-source AV) Wazuh (SIEM / EDR) AV-TEST published results AV-Comparatives
  • 2
    Write down your five evaluation criteria first — before looking at any results. Possible criteria: detection rate, false positive rate, performance overhead, ease of deployment, update frequency, logging/reporting, platform coverage, community support.
  • 3
    Gather evidence for each criterion — a published test result, a piece of documentation, a community post, or your own measurement. Record the source for every data point.
  • 4
    Score and recommend. Assign each criterion a weight (they need not be equal). Score the tool 1–5 per criterion. Compute a weighted score. Write a two-paragraph recommendation: would you deploy this tool in a small organisation? What are the caveats?
💡 Evaluation Template A simple approach: create a 5-row table (one row per criterion) with columns for Criterion, Weight (%), Score (1–5), Weighted Score, and Evidence Source. Sum the weighted scores for your final rating.
Part 1b

Reflection & Report

WEIGHT × 1

Your written deliverable is a maximum 4-page A4 report addressed to a hypothetical IT manager at a university department that currently has no formal security software stack. Written in clear, professional English (or French, if your programme permits) — it must not read as a product advertisement.

Page 1

Executive Summary & Context

Context paragraph (what the department runs, what data it holds, threat surface assumptions) + one-paragraph overall recommendation.

Page 2

Lab Findings

Key results from Exercises A, B, C — synthesised, not raw output. Use a table where appropriate. Interpret everything.

Page 3

Recommended Software Stack

One tool (or class of tool) per category from §1.2 + one-sentence justification. At least 2 must be tools researched fresh this week.

Page 4

Limitations & Surprises

Three limitations of your stack; non-software controls that fill the gaps; one paragraph on what surprised you most.

⚡ Academic Integrity Two students scanning the same VM will see identical Nmap output. Their reports must differ substantially in interpretation, finding selection, and recommendations. Identical or near-identical reports, regardless of shared lab environment, will be reviewed under the university's academic integrity policy.
